4.MySQL性能优化

发表于 2016-05-22   |  

简述

在现在的web系统中,数据库的操作越来越成为一个重要的限制因素,我们在这里探讨部分优化技巧,欢迎大家补充.

TIPS

1.为查询优化你的查询

大多数的MySQL服务器开启了查询缓存,就是说,当相同查询被执行多次以后,这些查询结果会放到一个缓存里面,这样后续相同的查询就不同操作表了

类似于CURDATE(),NOW()和RAND()这些类似的SQL函数不会开启查询缓存,因此我们需要使用变量来代替,从而开启缓存

2.Explain你的Select

在特定字段加索引,会加快查询

3.一行数据的时候使用LIMIT1

当你查询表的有些时候，你已经知道结果只会有一条结果，单因为你可能需要去fetch游标，或是你也许会去检查返回的记录数。

在这种情况下，加上LIMIT 1 可以增加性能。这样一样， MySQL数据库引擎会在找到一条数据后停止搜索，而不是继续往后查找下一条符合记录的数据。

6.避免select *

养成一个需要什么就取什么的习惯

7.为所有的表设置一个自增的ID主键

即使表中有一个唯一的字段,使用varchar来作主键会使性能下降

关于自增ID和UUID的好坏:

在单库中,使用自增id较快,如果要考虑到数据库集群,自增id可能产生冲突(一个解决方法,主用单id,从用双id),因此需要尽早考虑是否使用uuid

自增id的好处是:查询快,占用空间小

uuid占用空间较大,但是唯一

8.特定字段使用ENUM

ENUM 类型是非常快和紧凑的。在实际上，其保存的是 TINYINT，但其外表上显示为字符串。这样一来，用这个字段来做一些选项列表变得相当的完美。

在使用PHP连接数据库时,谨慎使用ENUM,因为PHP是弱类型,无法分辨1,’1’

9.尽可能使用NOT NULL

NULL也需要空间,并且在进行比较的时候,程序会变得更复杂,否则,尽量让你的子弹保持not null

10.IP地址尽量存为Unsigned Int

往往我们使用varchar而不是整型来存储IP,如果使用整型,只需4个字节,并且会带来查询上的优势.

11.固定长度的表查询得更快

并且，固定长度的表也更容易被缓存和重建。不过，唯一的副作用是，固定长度的字段会浪费一些空间，因为定长的字段无论你用不用，他都是要分配那么多的空间。

12.垂直分割

将数据库中的一张表按列编程几张表,这样可以降低表的复杂度和字段的数目,从而达到优化的目的;

将经常使用的数据放到一张表中;将经常需要更改的数据和需要查询的数据分开,以便更好的使用缓存;

注意,分割开的字段,如果经常需要join,会比不分割具有更差的性能

13.拆分大的delete或者Insert语句

当你需要执行这两种SQL时,你应该注意,这两种操作会锁表,肯呢个导致网站停止;

使用LImit,每次只删1000条

14.列越小,速度越快

对大多数引擎来,硬盘操作可能是其瓶颈

如果一个表只会有几列罢了（比如说字典表，配置表），那么，我们就没有理由使用 INT 来做主键，使用 MEDIUMINT, SMALLINT 或是更小的 TINYINT 会更经济一些。如果你不需要记录时间，使用 DATE 要比 DATETIME 好得多。

当然，你也需要留够足够的扩展空间，不然，你日后来干这个事，你会死的很难看，参看Slashdot的例子（2009年11月06日），一个简单的ALTER TABLE语句花了3个多小时，因为里面有一千六百万条数据。

15.选择一个正确的存储引擎

MyISAM 适合于一些需要大量查询的应用，但其对于有大量写操作并不是很好。甚至你只是需要update一个字段，整个表都会被锁起来，而别的进程，就算是读进程都无法操作直到读操作完成。另外，MyISAM 对于 SELECT COUNT(*) 这类的计算是超快无比的。

InnoDB 的趋势会是一个非常复杂的存储引擎，对于一些小的应用，它会比 MyISAM 还慢。他是它支持“行锁” ，于是在写操作比较多的时候，会更优秀。并且，他还支持更多的高级应用，比如：事务。

参见MySQL两种常见存储引擎

17.使用Prepared Statements

Prepared Statements可以检查一些你绑定好的变量，这样可以保护你的程序不会受到“SQL注入式”攻击。当然，你也可以手动地检查你的这些变量，然而，手动的检查容易出问题，而且很经常会被程序员忘了。当我们使用一些framework或是ORM的时候，这样的问题会好一些。

在性能方面，当一个相同的查询被使用多次的时候，这会为你带来可观的性能优势。你可以给这些Prepared Statements定义一些参数，而MySQL只会解析一次。
